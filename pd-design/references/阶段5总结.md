# 阶段5：总结与经验沉淀

## 角色
产品设计复盘师，客观记录项目经验，识别可复用的设计决策与改进点。

## 前置条件
阶段4（PRD生成与文档校验）已完成并通过校验。

## 步骤一：生成 Lessons.md（强制产出物）

在项目输出目录下生成 `Lessons.md`：

```markdown
# [项目名称] 经验教训记录

## 项目信息
- 项目名称：
- 完成日期：
- 项目类型：新系统设计 / 新功能模块 / 优化已有功能
- 项目周期：

## 做得好的地方
（至少3条，基于阶段1-4的实际产出过程）

## 需要改进的地方
（至少3条，诚实记录问题点）

## 关键决策记录
| 决策点 | 当时选择 | 依据 | 回头看 |
|--------|----------|------|--------|
| 审批流层级 | 2级审批 | 成本与风险平衡 | 是否合理？ |
| 表单字段 | 精简至8个 | B端用户效率优先 | 是否有字段遗漏？ |

## 需求/规则变化记录
| 环节 | 初始描述 | 最终确定 | 变化原因 |
|------|----------|----------|----------|
| | | | |

## 通用可复用经验
（提炼可复用于其他项目的经验，至少2条）

## 附录
- 产出物路径：
- 相关文档：
- Lessons.md 生成时间：
```

## 步骤二：更新全局计数器

读取并更新 `skills/pd-design/project-counter.json`：

```json
{
  "total_projects": +1,
  "last_lessons": [当前项目记录],
  "last_5_summary": null
}
```

## 步骤三：触发5项目汇总流程（条件触发）

当 `total_projects % 5 == 0` 时，**必须**执行：

1. 读取最近5个 `Lessons.md`
2. 在 `skills/pd-design/5-project-review/` 下生成汇总报告 `[N]th-batch-review.md`
3. 向用户汇报汇总结果，询问是否更新 Skills

### 汇总报告格式

```markdown
# 第N批项目经验汇总报告
（覆盖项目编号：#(N-4) ~ #N）

生成日期：
汇总人：

## 高频问题模式
| 问题 | 出现次数 | 典型案例 | 建议应对 |
|------|----------|----------|----------|
| | | | |

## 有效设计决策
- （按出现频次排序）

## 技能改进建议
| 优先级 | 改进点 | 具体建议 |
|--------|--------|----------|
| P1 | | |
| P2 | | |

## 是否更新Skills（用户决策）
- [ ] 是 — 更新以下内容：
- [ ] 否 — 积累更多样本后再说

用户确认后：

> 📋 第N批项目汇总完成。建议将以下改进点更新到 Skills 中：
> - P1: ...
> - P2: ...
> 
> 请确认是否执行更新？（是/否/修改后执行）
```

## 步骤四：检查是否需要更新现有 Skills

基于 Lessons.md 中的"需要改进的地方"，判断是否需要在本次项目后主动提出技能改进建议：

- 若有流程性缺陷（如某个步骤总是导致返工），在汇总报告中作为 P1 建议提出
- 若有模板性缺陷（如某类表格格式不适用），直接更新对应 references 文件

## 最终交付汇报格式

```
✅ 阶段5「总结」执行完毕
   - Lessons.md 已生成：[路径]
   - 项目计数器已更新：累计 N 个项目
   - 5项目汇总：[已触发 / 距下次还有 (5-N%5) 个项目]
   - 进度文件已归档：[路径]
```

## 严格约束
- **必须生成 Lessons.md**：即使项目失败或中途终止，也应记录已执行过程的经验
- **诚实记录**：不美化问题，"做得好的地方"和"需要改进的地方"均需≥3条
- **5项目汇总强制触发**：不可跳过，不可延迟到下次
- **计数器持久化**：`project-counter.json` 必须在项目完成后立即更新，不得丢失
