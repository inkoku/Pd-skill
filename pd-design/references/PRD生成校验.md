# 阶段4：PRD生成与文档校验

## 角色
资深产品文档工程师，基于业务建模和原型输出，生成结构完整、逻辑严谨的 PRD 文档，并进行自校验。

## 输入
- 需求分析报告（阶段1产出）
- 业务模型（ER图、状态机、时序图、用例图，阶段2产出）
- 页面功能清单（阶段2产出）
- 高保真原型HTML（阶段3产出）
- 业务词汇表/数据字典

## 文档校验清单

生成 PRD 后，严格按以下清单逐项校验：

### 结构完整性
- [ ] 包含文档信息表（版本、作者、日期、状态）
- [ ] 包含概述（背景、目标、范围、术语表）
- [ ] 包含用户与角色矩阵
- [ ] 包含业务流程
- [ ] 包含所有功能模块的详细说明
- [ ] 包含数据字典
- [ ] 包含非功能需求（性能、安全、可用性等）
- [ ] 包含风险与依赖分析
- [ ] 包含变更记录

### 内容一致性
- [ ] 功能清单中的字段与原型页面一致
- [ ] 用例规约中的前置/后置条件与功能描述一致
- [ ] 状态机中的状态与功能描述中的状态一致
- [ ] 泳道图中的角色与角色矩阵一致

### 质量检查
- [ ] 无拼写错误和语法错误
- [ ] 术语使用一致（同一概念只用一个词）
- [ ] 量化的描述可测量（无"快速"、"便捷"等模糊词）
- [ ] 每个功能有明确的优先级（Must/Should/Could）
- [ ] 每个风险有明确的应对措施

### 可读性
- [ ] 标题层级清晰（无跳级）
- [ ] 表格有表头
- [ ] 页面功能清单有角色归属

## 校验报告格式

```
📋 PRD 文档校验报告

✅ 通过项 (N项)
- [ ] xxx

⚠️ 需修正项 (N项)
- [ ] 缺失项：xxx（在 5.2.3 节补充字段 xxx）
- [ ] 不一致：原型中"金额"字段类型为文本，与数据字典中"数字"不一致
- [ ] 模糊描述：5.2.1节"响应要快"→ 建议改为"P95 < 2秒"
- [ ] 术语不一致："订单"与"采购单"混用 → 统一为"采购申请单"

❌ 严重问题 (N项)
- [ ] xxx

📌 请逐项确认修正后，再次提交校验。
```

> **注意**：校验未通过时，必须修正后再输出最终版本，不可跳过校验步骤。

## PRD 保存

- 输出文件命名：`[项目名称]PRD_v[版本号]_[日期].md`
- 同时转换为 `.doc` 文件
- 更新进度文件 `.prd-progress.json`
